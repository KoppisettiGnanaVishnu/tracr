# Tracr

**Tracking Review And Code-origin Record**

**Live demo:** https://tracr-6gyu.onrender.com
*(hosted on free-tier infrastructure — first load can take 30–50 seconds if the service has been idle, since the free tier sleeps after inactivity)*

An accountability layer for AI-assisted software development.

---

## The Problem

AI coding assistants — GitHub Copilot, Claude, Cursor — are now a standard part of how developers write code. A large and growing share of production code starts out as an AI suggestion rather than something typed by hand. This shift has made development faster, but it has introduced a problem that most teams haven't built any process around yet.

The core issue isn't that AI writes *worse* code than humans across the board. It's more specific and, in some ways, more dangerous: **AI-generated code tends to receive less scrutiny during review than human-written code, simply because it looks polished and confident.** A reviewer skimming a clean, well-formatted function is far less likely to slow down and check it carefully than they would for a messy or unfamiliar piece of code — even though "looks clean" and "is actually correct" are not the same thing.

This isn't a hypothetical concern. Independent research backs it up:

- An analysis of over 300,000 AI-authored commits across more than 6,000 public GitHub repositories found that AI coding tools introduce nearly **twice as many security issues as they resolve**.
- In a controlled study, developers using AI tools took measurably **longer** to complete tasks than those who didn't — yet still believed afterward that AI had made them faster. That gap between perceived and actual reliability is exactly the kind of overconfidence that lets risky code slip through unchecked.
- Studies on AI-generated pull requests specifically found that human reviewers are **less critical** of AI-written code, allowing redundant logic and hidden design issues to pass review more easily than they would in human-authored code.

So the real gap isn't in how AI writes code — it's in how carefully that code gets checked afterward. No common code review tool currently tracks *where* code came from or flags when AI-generated code in a sensitive area has gone unreviewed. That's the specific gap Tracr is built to fill.

## A Concrete Example

**Without Tracr:**

A backend engineer, Priya, is reviewing a pull request from a teammate. The PR includes a new `refundPayment()` function, written mostly by Copilot. It's well-formatted, uses sensible variable names, and reads like something Priya might have written herself. She's got several other PRs waiting in her queue, so she skims it, sees nothing obviously wrong, and approves it within a minute.

Three weeks later, a customer is refunded twice for the same order. The bug traces back to that exact function — a subtle logic error in how it checked for duplicate refund requests. Nobody had actually read it closely, because it looked too clean to warrant a second look.

**With Tracr:**

The moment that PR is opened, Tracr automatically scans the diff and posts a comment directly on the pull request:

> ⚠️ **Tracr risk warning**
> This PR contains changes that touch sensitive areas:
> - **payment.js** — payment handling, auth token handling
>
> Please ensure these changes have been reviewed by a human before merging.

Priya now sees this warning in the exact place she's already looking, before she clicks approve. Tracr doesn't replace her judgment, block the merge, or fix the code — it simply makes sure she doesn't skip the one function that actually needed a closer look, at the exact moment her guard would otherwise be lowest.

## The Solution

Tracr is an accountability layer that sits alongside the existing AI-assisted development workflow. It does four things:

1. **Tracks code origin** — records whether a given piece of code was written by a human or generated by an AI tool.
2. **Scans for risk** — checks AI-generated code for patterns indicating sensitive functionality: authentication, payment handling, database deletion, environment variables and secrets.
3. **Surfaces warnings where they matter** — when AI-generated code in a risky area hasn't been reviewed by a human, it posts a warning directly on the GitHub pull request, automatically, before merge.
4. **Builds a trust score over time** — a 0–100 score per repository, calculated from real tracked data: percentage of AI-generated code, risk rate, bug count, and revert history. Not a guess — a number computed from whatever has actually been recorded.

Importantly, Tracr does not try to make AI write better code, and it does not try to detect AI-written code by guessing at writing style — both of those are open, unsolved problems industry-wide (similar in spirit to the well-known unreliability of AI-text detectors). Instead, Tracr accepts honest origin tagging as input and focuses its engineering effort on the part of the problem that's actually solvable today: making sure the *review* step catches what it currently misses.

## Workflow
Code is written  →  tagged as AI or human  →  Pull Request opened

↓

Risk scanner checks the diff for sensitive patterns

↓

If AI-written code touches a sensitive area and is unreviewed →

Tracr posts a warning comment directly on the PR

↓

Human reviewer sees the warning before approving

↓

Trust score for the repository updates based on real, accumulated data

In more detail:

- **Capture:** Each code block is tagged as AI-generated or human-written. In this MVP, tagging happens via a direct API call rather than automatic detection — building a fully automatic, reliable detector is a substantial problem on its own (see `pocDetector.js` for an early proof-of-concept using typing-speed analysis, which reliably distinguishes typed text from pasted/inserted text, though it can't yet distinguish an AI suggestion from a manual paste).
- **Risk scan:** A transparent, rule-based scanner checks code against a set of sensitive keyword patterns (auth, payment, password, token, database deletion, environment variables). Kept deliberately simple so the logic is easy to explain and verify, rather than a opaque black-box security scanner.
- **GitHub integration:** Using the GitHub REST API (via Octokit), Tracr pulls a pull request's diff, runs the scanner against the added lines, and posts a comment back on the PR if anything risky and unreviewed is found.
- **Trust score:** Computed with a weighted formula — risk rate counts for the most (40%), since unreviewed risk is the core problem; bug count and revert history each contribute the rest, since they're more downstream signals of code quality.

## Try It Yourself

The live deployment is real and responds to actual requests — not just a static demo page.

```bash
curl -X POST https://tracr-6gyu.onrender.com/api/code-block \
  -H "Content-Type: application/json" \
  -d '{"repo_name":"demo-repo","file_path":"login.js","code_snippet":"function login(password, token){}","is_ai_generated":true}'
```

Refresh https://tracr-6gyu.onrender.com afterward — you should see the new entry tagged `AI` and `RISKY`, with the trust score updated accordingly.

## Tech Stack

| Layer | Technology |
|---|---|
| Backend | Node.js, Express |
| Database | SQLite (via better-sqlite3) |
| GitHub integration | GitHub REST API, via Octokit |
| Frontend / dashboard | Plain HTML, CSS, and JavaScript — no framework |
| Deployment | Render (Singapore region), free tier |
| Uptime | Kept warm via UptimeRobot, pinging every 5 minutes |

The stack was kept deliberately simple and dependency-light — no React, no ORM, no heavyweight framework — so that every piece of the system is easy to read, explain, and verify line by line.

---

## Architecture Note: Application, Not Extension

Tracr is built as an application, not a browser or editor extension. A VS Code extension for automatic AI-vs-human code capture was considered and partially prototyped (`pocDetector.js`), but was scoped out of this MVP — Copilot and similar tools don't expose a public "suggestion accepted" event, making reliable automatic capture a substantial project in its own right. The engineering effort for this version went instead into the risk-scanning logic, GitHub integration, and trust-score calculation, with manual tagging used as an honest, 100%-accurate stand-in for automatic capture.

## Honest Limitations

- **Risk detection** is keyword/pattern-based, not a full static-analysis security scanner. This was a deliberate choice to keep the logic transparent and easy to verify, not a claim that it's the most sophisticated possible approach.
- **Code-origin tracking is forward-looking only** — it cannot retroactively determine whether code already sitting in an existing repository was AI-generated or human-written.
- **Bug and revert counts are currently entered manually**, rather than pulled automatically from git history — a natural next step.
- **PR comments currently post under the developer's own GitHub account** (since the API call authenticates with a personal token), rather than a separate bot identity. A dedicated GitHub App would be the next step toward a more polished, team-ready version.
- A comparable commercial tool, [Git AI](https://github.com/git-ai-project), tackles an overlapping problem — line-level AI code attribution across the full development lifecycle. Tracr is intentionally narrower in scope, focused specifically on review-time risk warnings and trust scoring rather than full attribution tracking.
